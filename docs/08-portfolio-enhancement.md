# 포트폴리오 강화 가이드 (2026-03-10 기준)

## 1. 목적
`필지랩`을 단순 CRUD/조회 프로젝트가 아니라, 실제 운영형 정밀 분석 제품 포트폴리오로 보이게 만드는 정리 기준이다.

현재 포지셔닝:
- 공공 토지정보 조회 서비스와 상위 사업성 분석 플랫폼 사이의 `정확도 중심 작업 시스템`
- 핵심 차별점은 `지도 워크스페이스`, `구역 분석`, `저장/비교`, `파일 비동기 처리`, `건축물 지표`, `AI 보조 검토`

## 2. 현재 포트폴리오에서 이미 강한 부분
- 조회 3축 구현: 개별조회, 지도조회, 파일조회
- 지도 기반 구역 분석: PostGIS 교차 계산, 90% 규칙, score 기반 분류, 저장/재열람/비교
- 정확도 설명: `overlap_ratio`, `confidence_score`, `ai_recommendation`, `anomaly_level`
- 건축물 지표 결합: 노후도, 평균 사용승인년도, 총연면적, 총대지면적, 평균 용적률, 과소필지 비율
- 운영 설계: VWorld direct 실패 시 고정 IP 프록시 fallback
- 제품 완성도: 인증, 기록, 정책 페이지, Android wrapper까지 연결

## 3. 지금 부족한 포트폴리오 증빙
### 3.1 성능 증빙
아직 문서상 계획 대비 실제 수치 캡처가 없다.

필수 산출물:
- Redis 캐시 hit/miss 비교
- Bulk 처리 전후 샘플 작업 시간
- 구역 분석 응답 시간 샘플

권장 저장 위치:
- `docs/assets/perf-cache-compare.png`
- `docs/assets/perf-bulk-sample.png`
- `docs/assets/perf-zone-analysis.png`

### 3.2 운영 증빙
아직 코드 대비 운영 관제 증빙이 약하다.

보강 권장:
- Sentry 연결 여부 문서화
- `api_error_rate`
- `vworld_direct_failure_rate`
- `vworld_proxy_usage_rate`
- bulk queue backlog 또는 worker health 체크 방식

### 3.3 사용자 시나리오 문서
현재 `기능 설명`은 강하지만 `누가 왜 쓰는가` 문서는 없다.

최소 페르소나:
1. 공인중개사
2. 정비/재개발 분석 담당자
3. 토지 투자 판단 사용자

문서에 반드시 들어갈 것:
- 문제 상황
- 입력 방식
- 필지랩 사용 흐름
- 기존 방식 대비 빨라진 점
- 정확도/재검토 측면에서 나아진 점

### 3.4 검증 자동화
현재 코드베이스는 기능 폭 대비 테스트 흔적이 약하다.

포트폴리오 관점에서 큰 보강 포인트:
- API smoke test
- 핵심 시나리오 QA checklist
- 배포 전 회귀 점검 절차

## 4. 면접에서 강조할 기술 포인트
- `Next.js + FastAPI + PostgreSQL/PostGIS + Redis` 조합으로 지도/비동기/인증/저장 흐름을 한 제품으로 엮었다는 점
- 외부 공공 API 불안정성에 대응해 `direct -> proxy fallback` 경로를 설계했다는 점
- 단순 조회가 아니라 `저장`, `비교`, `리뷰 큐`, `정확도 설명`까지 제품 흐름으로 확장했다는 점
- AI를 계산 엔진이 아니라 `추천/설명/이상치 검토 보조`로 제한한 점

## 5. 자기소개서/이력서 문장 방향
- "공공데이터 조회 서비스를 만들었다"보다 "필지/구역 검토 워크플로우를 제품 형태로 설계하고 구현했다"
- "AI를 붙였다"보다 "결정론적 계산 엔진 위에 추천/이상치 보조 계층을 설계했다"
- "지도 기능을 넣었다"보다 "저장/비교/재열람이 가능한 map-first workspace를 설계했다"
